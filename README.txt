0. 不依赖 PowerShell 直接碰中文文件名，可以试试spreadsheet技能。

1. 先到**来源中的指标清单**获取【需要测试的报表】sheet1中【需要测试的指标】的**计算逻辑**，并优先确认：

   * 还需要用户补充哪些台账等；
   * 是单月还是累计；
   * 是否排除 **D类已撤场**；
   * 是否排除 **非考核项目**；
   * 是否有**区域级 / 空间级 / 条线级附加项**；
   * 高维是否允许直接由项目维度汇总（维度也称为级）。
   * **查指标清单时，必须同时核对“序号列 + 指标名称 + Excel原始行号”三者，不得只看 pandas 过滤结果的行号。** 例如用户说“读取序号9行”，优先理解为“序号=9”这一条，而不是 DataFrame 的第9行或 Excel 物理第9行；必要时直接用 openpyxl 读取原始单元格复核。

2. 再运用来源或对话中的其他台账表格测试该指标是否数值正确；永远忠于指标清单，永远不要预设指标能测通过。

3. **【非考核项目台账】和【项目查询】是基础辅助台账：**

   * 【非考核项目台账】用于判断是否为非考核项目；**在该台账内的是非考核项目，不在的是考核项目**。
   * 【项目查询】用于判断项目属性，如项目编码、项目等级、项目状态、是否已撤场等；例如 **“D类已撤场” = 项目属性 D + 项目状态 已撤场**。这个台账很关键，因为部分报表没有项目编码，但项目级的一切数值原则上都必须通过编码匹配，因此需要通过此台账连接其他需要的台账。

4. 对**非项目维度**指标，所谓“指标库”一般优先指该报表的**项目维度报表**，这种情况下：

   * 如果**项目维度已证实正确**，则**直接从项目维度报表汇总**测试高维；
   * **不要默认从源台账重新计算高维**；
   * 只有当指标清单明确写了高维还有额外逻辑时，才在项目汇总基础上叠加这些规则测试。

5. 如果**项目维度测试未通过**，则**缓测高维度并汇报**，不要继续往上验证。

6. 测试高维前，必须先从指标清单确认该维度是否需要：

   * 排除 **D类已撤场**；
   * 排除 **非考核项目**；
   * 增加 **区域级A / 空间级A / 调整项B** 等特殊逻辑。
     **不能通过报表差异倒推这些条件。**

7. 对累计类报表，先确认口径是：

   * 直接取报表月当月值；
   * 还是按 **1~报表月逐月累计**。
     不能因为报表是年度累计就默认“乘12”，也不能因为是12月报表就默认“只取12月”。

8. 有**查不到、字段对不上、逻辑缺项、枚举值不一致**的情况，要先提问确认，用户会给予补充或引导。

   * 反面教训：这次漏读 `带资摊销计划数_项目`，根因不是“原表没有定义行”，而是**把结果指标名当成基础定义名去检索**。报表里是 `带资摊销调整计划数`，指标清单单独定义行写的是 `带资摊销计划数_项目`，名称并不完全一致。以后查指标清单时，先做“结果指标名 -> 基础定义名/项目定义名”的映射，再按 **序号列 + 指标名称 + Excel原始行号** 三重校验；不能只按结果名模糊搜索后就下结论。

9. python测试脚本保留在该文件夹，以后每次测试就不用再生成一次了，并且持续优化迭代，并用中文写注释。不要每次测试都建一个新的脚本，而是在现有脚本上优化；目标是省token。

10. **三大摊销（带资 / 智能化 / 质效）测试方法：**

   * 优先使用《1 摊销模板-新 - 2025年四季度v1.5(去重版) - 导入整理 V2》或类似文件名。
   * 其中 `B-按项目统计` sheet 是三大摊销的项目级底表。
   * 先取每个项目的**还原金额**（即 `发生数 - 计划数`）作为默认权威对照值。
   * 默认先验证 **发生数 - 计划数 = 还原金额**，而不是分别强行核对计划数、发生数。
   * 如需深挖，再沿着对应单元格公式追到源底表。
